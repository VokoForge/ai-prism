# Day 14 · AI Agent 现状 2026 H1: 5 大场景 16 个项目图谱

> **日期:** 2026-05-29 · **类型:** 景观 · **难度:** 各级别
> **阅读时间:** ~14 分钟

我们一年发两次"现状图"。这是 2026 H1 版。挑了** 16 个有代表性的项目**横跨** 5 个场景**, 干活的开发者至少该知道这些名字。目标不是排名 (Day 09 和 13 排了) — 目标是给你一张心理地图, 下次有人扔个名字, 你知道往哪放。

## 怎么挑的

三个过滤器, 按顺序:

1. **2025-2026 活跃** — 过去 6 个月至少一次重要发布。
2. **>1k GitHub stars 或 >1k PyPI/npm 周下载** — 实用性经过验证, 不是炒作。
3. **设计点不同** — 我们不放 4 个 langgraph fork。

下面列出 16 个。我们故意大小混搭: 小项目才是设计教训在的地方。

![agents-landscape](assets/day-14-agents-landscape.svg)

## 5 大场景

```
┌────────────────────┐  ┌────────────────────┐  ┌────────────────────┐
│ 1. 本地执行        │  │ 2. 自主任务        │  │ 3. 多智能体        │
│   (单用户,         │  │   (长时域,         │  │   (专家团队,       │
│    单机器)         │  │    自我驱动)       │  │    角色分工)       │
└────────────────────┘  └────────────────────┘  └────────────────────┘
┌────────────────────┐  ┌────────────────────┐
│ 4. 低代码 /        │  │ 5. 垂直 /          │
│    可视化          │  │    领域专用        │
│    (零代码或       │  │    (写作/代码/     │
│     极少代码)      │  │     研究…)         │
└────────────────────┘  └────────────────────┘
```

为什么 5 个? 这是最小的切分, 每个场景至少 3 个代表, 设计上明确不同。4 个场景会把"自主"和"多智能体"合并丢教训; 6 个场景会把"垂直"硬切。

---

## 场景 1 — 本地执行 (4 个项目)

"跑在你笔记本上"那一档。用户是 agent 唯一的利益相关者。

| 项目 | 一句话 | 为什么重要 |
|------|--------|------------|
| **OpenHands** | Docker 里跑的沙箱 dev agent | 主流框架里唯一把代码执行当安全边界的 |
| **Aider** | 终端里配对的 AI 编程 | "好的配对编程 UX 长什么样"的标杆 |
| **Continue** | VS Code / JetBrains 插件 | 最可扩展的 IDE 侧 agent (开源, MCP 原生) |
| **Cline** | 带 MCP 的 VS Code 插件 | IDE 圈最激进的 MCP 拥抱者 |

**这档的教训:** 本地 agent 赢在**延迟和信任** — 用户看到每个动作, 沙箱是一个容器, 循环紧凑 (亚秒级每次工具调用)。天花板是"一个人一节会能审完的东西"。

## 场景 2 — 自主任务 (3 个项目)

"我给你个目标, 你 2 小时后回来"那一档。

| 项目 | 一句话 | 为什么重要 |
|------|--------|------------|
| **OpenAI Deep Research** | 多步研究, 引用链 | 证明了"长时域 + 工具用 + 验证"能规模化 |
| **Manus** | 通才云 agent (闭测) | Deep Research 的中国答案; 第一个把"computer use"真正部署上线的 |
| **Forge** (vieko) | 结果驱动的编排 | 把成功定义为*规范*而不是流程 — 我们见过最干净的契约 |

**这档的教训:** 设计瓶颈已经不是 LLM; 是**验证循环**。Forge 的"spec 作为契约, agent 决定怎么做"模式是最有希望的方向。Manus 告诉世界 computer-use 是可部署的 (有 caveat)。

## 场景 3 — 多智能体 (3 个项目)

"专家团队"那一档。角色, 交接, 有时冲突。

| 项目 | 一句话 | 为什么重要 |
|------|--------|------------|
| **LangGraph** | 基于图的编排 | 行业标准; "agent 图的 k8s" |
| **CrewAI** | 角色制 crew | 学习曲线最低; 对 agent-as-person 最有主张 |
| **AutoGen** (Microsoft) | 对话优先 | OpenTelemetry 故事最好; 学术研究最友好 |

**这档的教训:** "多智能体" ≠ "更多 agent"。赢在**清晰的角色边界**和**显式交接协议**, 不是堆更多 LLM。3 个框架只在这两个轴上分叉; 剩下的都是实现细节。

## 场景 4 — 低代码 / 可视化 (3 个项目)

"非开发者能搭"那一档。

| 项目 | 一句话 | 为什么重要 |
|------|--------|------------|
| **n8n** | 带 AI 节点的工作流自动化 | 70k+ stars; 原生 LLM 步骤的事实 OSS 替代 Zapier/Make |
| **Flowise** | 可视化 LangChain 构造器 | 一个下午搭原型 RAG 应用的最佳选择 |
| **Dify** | LLMOps 平台带可视化流水线 | 中国市场最强; 出厂带 BAAI embeddings |

**这档的教训:** 可视化构造器**没**干掉框架开发。它们服务**非开发者构建者** (PM, 设计师, 运维), 同样的工作流到生产往往用代码重建。把它们当"设计工具"阶段, 不是"部署"阶段。

## 场景 5 — 垂直 / 领域专用 (3 个项目)

"做好一件事"的 agent 那一档。最热的增长区。

| 项目 | 一句话 | 为什么重要 |
|------|--------|------------|
| **StoryForge** | 带风格 DNA 的长篇小说写作 | 最成熟的 7 阶段创意工作流; 52 种经典作家风格 |
| **NOVIX** | 上下文工程的多智能体小说写作 | 长篇连续性最佳; 同人创作支持 Wiki 导入 |
| **DeepResearch 类** | "读 50 篇论文, 写综述" | 第一个用户超 100 万的垂直; 边还毛糙 |

**这档的教训:** agent 的未来是**垂直**的, 不是横向的。通才 (Day 13) 提供基底; 垂直 (Day 14) 是价值被拿走的地方。预期 2026-2027 每一个"AI X"创业都是垂直 agent 加一个薄薄的通才壳。

## 跨场景观察

### 1. MCP 现在是入场券

**16 个里 13 个**支持 MCP, 要么原生, 要么一适配器。3 个不支持的 (OpenAI Deep Research, Manus, Aider) 都在路线图上。2025 的协议之争 (MCP vs OpenAI function-calling vs Anthropic tool-use) 结束了: **MCP 赢了本地工具层**。Function-calling 仍是托管 API 的线协议。

### 2. Skills 是新插件

**16 个里 8 个**有"skills"或"extensions"机制 (LangGraph tools, CrewAI tools, Continue 的斜杠命令, StoryForge 方法论插件)。Skills 胜过 raw code-as-config, 因为它们**模型可读** — 模型加载 skill 描述, 决定什么时候用。2023 函数胜 raw strings 的转变, 现在发生在包级别。

### 3. 记忆是未解决的问题

16 个里只有 **Letta, StoryForge, NOVIX** 对"跨 session 记忆"有认真的答案。其他都重置。行业会在 2026 H2 解决这个问题, 但现在你想要 30 天对话, 你得自己搭。

### 4. 中国生态在 UX 上领先

**Manus, Dify, StoryForge, data-lens-skill 生态** 全部交付了英语圈没匹配的 UX。一部分因为中国用户更早接受复杂度 (MCP, 多智能体), 一部分因为中国 LLM 栈 (Qwen, DeepSeek, Doubao) 一直激进地为 agent 调优。

## 我们排除的 (和原因)

- **Bolt, v0, Lovable** — 私有模型栈的包装, 设计教训不多。
- **Devin** — 闭源, 没法审内部。
- **Smol-developer, AutoGPT (原版)** — 抛弃或维护模式, 不代表 2026。
- **BabyAGI** — 历史重要, 设计已被超越。
- **GenericAgent** — 上下文工程强, 采用度低, 留观。

## 我们的看法

如果你** 2026.05 开始新 agent 项目**, 路径是: 选个垂直 (场景 5), 搭在通才基底上 (场景 3), 通过 MCP 暴露以便任何 host 都能用, 发 Skills 让模型能好用, 把记忆当护城河。横向 agent 会商品化; 有专有数据 + 记忆的垂直不会。
